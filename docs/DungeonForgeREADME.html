READ THIS BEFORE YOU TOUCH IT

DungeonForge is in pre-alpha. The core gameplay loop has not been vetted. Subsystems exist, they wire together, and on a good day the whole thing boots — but nothing is balanced, almost everything has rough edges, and which thing snaps next is genuinely a coin flip.

I'm not pretending otherwise. Right now this is for two kinds of people: folks who want to render starter content and poke holes in the systems with me, and folks who want to sit back, watch it develop, and join when it looks like it won't hurt them. Both are valid. Pick your role below.

##What the hell is this

DungeonForge is a self-authored dungeon crawler where nothing is pre-drawn. There are no built-in sprites, no built-in items, no built-in enemies. You describe a thing in plain English — "a cinder-veined wyrm with brass scales" — and an AI rendering pipeline turns it into a playable sprite with walk frames, a back view, a corpse, and stats scaled to your dungeon's lore.

The world exists entirely in latent space until someone describes it. The engine, the render queue, and the LLM inference all run together and feed each other in real time. Rendering takes 1–5 minutes per asset — that's not a bug, that's the texture of the game. You configure while it renders. You re-enter dungeons while sprites stream in. The waiting is the rhythm.

##What it's meant to become

DungeonForge is a dungeon crawler today, but the shape underneath is a game engine wearing a dungeon crawler like a mask. The roadmap is building toward a place where players don't just play it — they fork their own game modes from inside the Forge.

01

A FORGE, NOT A GAME

The hub is a workshop ring — Terra Fabricator, Library, Machinarium, Arcanum, Substance Lab, Undercroft. Each is a different axis of world-building, and they all feed the same render queue.

02

EVERYTHING PLAYER-GENERATED

Creatures, weapons, props, lore, NPC personalities, drop pools — all authored by description and reified by render. No shipped assets. The engine refuses to assume what your world looks like.

03

EXPERIENCES, FORKABLE

A whole dungeon — lore, rules, rosters, triggers — is a single experience JSON doc. EXP: share codes encode the entire world to base64. .dpack bundles ship with sprites included.

04

NEW GAME MODES, INDEFINITELY

Past the dungeon-crawl, an isometric overworld view opens up RTS-shaped play. Past that, the engine stays open — the substrate is designed to keep accepting new modes without rewriting.

##Get started

You host your own instance. You bring a Graydient API key. The launcher does the rest.

Prerequisites

  • Python 3.11+ with pip
  • A modern Chromium-based browser (Three.js + ES modules + ImageBitmap)
  • A Graydient v3 API key — txt2img, img2vid, and LLM access
  • Patience for 1–5 min per first-time render. (Subsequent renders cache.)

Quickstart

bash · clone, install, launch ⎘ copy
# clone git clone https://github.com/UnlimitedEditing/DungeonForge cd DungeonForge # install pip install -r requirements.txt # launch (tkinter cross-platform launcher) python launcher.py → create a profile → paste your Graydient API key (Fernet-encrypted on disk) → click START # …or run the server directly uvicorn forge:app --reload --port 8000 # then visit http://localhost:8000

First boot, the Forge hub will be empty. That's correct. Open the Library, write some lore, and start a dungeon. The first render takes a minute or two — that's the texture of the game.

##Pre-emptive Q&A

Q : Is this a product?
No. It's a pre-alpha solo-dev project. I'm not selling anything. There's no roadmap to monetisation. It might never reach 1.0.
Q : Why is rendering so slow?
Because real generation takes real time. The whole game is designed around configure-while-it-renders. The slowness is the texture, not the friction.
Q : Can I run it without Graydient?
Not today. The render substrate is deliberately not backend-agnostic — that complexity isn't a fight I'm picking in pre-alpha. Maybe later.
Q : Is my API key safe?
Your password is bcrypt-hashed. Your API key is Fernet-encrypted at rest with a PBKDF2-derived key. The decrypted key only ever lives in the server's in-memory session map and is lost on restart. I don't log it. I don't cache it.
Q : Will my experiences survive an update?
Probably. Experiences are JSON and the schema is stable enough for forking. But this is pre-alpha — if the schema needs to evolve, it will, and I'll write migrations when I can.
Q : Can I contribute code?
Not the priority right now. The system is still finding its shape and PRs against a moving target are painful for everyone. Issues, however, are gold.

##One more thing

DungeonForge is the project I needed an excuse to build. It's slow, it's strange, it occasionally produces a wyrm with too many legs, and watching the first sprite of a dungeon resolve out of latent noise is genuinely one of my favourite feelings.

If that sounds like a thing you'd enjoy holding a torch for — welcome to the forge. If not, file is your cue to close the tab. No hard feelings.
